## Architecture

**Monsters of Jungle** is a static site for showcasing electronic music artists. It uses [Vituum](https://vituum.dev/) (Vite + Nunjucks) for page templating and a custom Vite plugin (`sj-web-crate`) as a build-time headless CMS.

### Content Pipeline (`plugins/sj-web-crate/`)

The core architectural pattern: Markdown files with YAML frontmatter in `/content/` are parsed at build time and exposed as Vite virtual modules.

- `parseCollection.ts` — reads `.md` files via `gray-matter`, validates required fields, derives slugs from filenames
- `generateDts.ts` — auto-generates `plugins/sj-web-crate/sj-web-crate.d.ts` for TypeScript safety
- `plugin.ts` — registers virtual modules, watches content for HMR, sorts entries by `gridOrder`

Collections are configured in `vite.config.ts`. Currently active: `artist` (from `/content/artists/`) and `upcoming-event` (from `/content/events/`).

If a collection has `pageTemplate` set, the plugin also generates one `src/pages/<name>/<slug>.json` per entry at `buildStart`. The Nunjucks plugin picks these up and renders them into `dist/<name>/<slug>/index.html`. These generated JSON files are gitignored.

### Content & Asset Organisation

Each collection follows a strict separation:

- `content/<collection>/` — Markdown files only (frontmatter metadata + body text). No images here.
- `public/<collection>/` — Image assets for that collection, referenced by filename in frontmatter.

For example, an event at `content/events/13-04-2026.md` has `flyer: MoJ-2026-02.png` in its frontmatter, and the image lives at `public/events/MoJ-2026-02.png`. This keeps image references explicit and predictable rather than inferred from slugs.

### Pages & Templating (`src/pages/`)

Nunjucks `.njk` templates. Pages import virtual modules and pass data to Lit-HTML components.

### Client-Side Rendering (`src/scripts/`, `src/components/`)

Lit-HTML (not Lit elements — just the `html` tagged template) for rendering lists and grids. Components like `render-artist-grid` and `render-mix-list` are pure render functions.

### Feature System (`src/scripts/crateFeatureResolver.ts`)

Artists can opt into features via the `features: ["radioShow"]` field in their frontmatter. The resolver returns dynamic attachment content. The `radioShow` feature calculates the next 4th-Saturday-of-month broadcast date.

### Deployment

GitHub Actions (`.github/workflows/deploy.yml`) builds and deploys to GitHub Pages on push to `main`. The site is served from the base path `/monstersofjungle.de/`.

## Architectural Direction: Static HTML Generation

The site is moving away from JavaScript-driven rendering toward fully pre-generated static HTML. Keep this in mind when adding features:

- **Avoid JavaScript for content rendering.** Currently `src/scripts/` uses Lit-HTML to render artist grids, mix lists, and event info at runtime. These should be migrated to Nunjucks templates rendered at build time.
- **Avoid URL search params for routing.** The current artist detail page uses `?slug=dylan` and reads it via JS. The target architecture is one `.html` file per artist (e.g., `artists/dylan.html`), generated by Vituum from a Nunjucks template + collection data.
- **New features should be build-time first.** If content is available at build time (it always is — it comes from the virtual module), render it in Nunjucks, not in a client-side script.

The `sj-web-crate` plugin already provides all data at build time. The JS layer exists for historical reasons and should shrink over time, not grow.

## Adding Content

### Artist

Create `content/artists/<slug>.md`:

```yaml
---
artistNameLabel: Artist Display Name
gridOrder: 10
mixLinks: []           # Array of { label, link }
features: []           # Optional: ["radioShow"]
attachment: null
---
```

### Event

Create `content/events/<date>.md` (e.g., `13-04-2026.md`):

```yaml
---
flyer: filename.png    # Must exist in public/events/
---
Event description text here.
```

Place the flyer image at `public/events/filename.png`.
